## 1. 범위 베이스라인 (calibrated "C")

"다 한다"도 "최소만 한다"도 아닌, 이 규모(7도구·stateless·3인·데드라인)에 맞춘 현업 베이스라인.

**기본값(DoD) — 레포 [CLAUDE.md](../../CLAUDE.md)가 이미 요구**: 타입힌트+docstring · annotations 5종 · `pytest`/`ruff` 통과 · 버전 3중 동기화.

| 항목 | 결정 | 형태 |
| --- | --- | --- |
| 공유 타입(=계약) | ✅ 하되 가볍게 | `models.py` 한 파일에 Pydantic 타입 4종, **1인 소유**. SCHEMA_VERSION·게이트키퍼·fixtures 1급화 ❌ |
| 테스트 | ✅ 얇게 | 도구별 정상응답 + e2e 체인 + annotations 존재. 광범위 매트릭스 ❌ |
| 결정적 엔진 | ✅ 필수 | `recommend_dresscode` 규칙 테이블 100% 결정적(LLM폴백 X), `parse_occasion`는 `reference_time` 인자 → `idempotentHint=true` 정직 |
| annotations | ✅ 필수 | 5종 정직하게(정적·결정적 = readOnly/idempotent=true, openWorld=false) |
| 배포 + CI | ✅ 필수 | 카카오 클라우드 streamable-http + 공개 URL + PlayMCP 등록, ruff/pytest 게이트 |
| 품질 카피 | ✅ 싼 승리 | 면책 문구 + 계절 슬롯. **장례·종교 시나리오는 복귀 후(v1)** |
| privacy 가드 | 🟡 MVP 최소 | 텍스트 입력만 → 이미지 입력 거부 + PII 미로깅. 본격화는 이미지 기능 시 |

## 2. MVP 도구 4개 + 공유 타입

| 도구 | 기능 | annotations(예상) |
| --- | --- | --- |
| `parse_occasion` | 자연어 상황 → 구조화(일시·장소·성격·제약·계절), `reference_time` 인자 | readOnly=T, idempotent=T, openWorld=F, destructive=F |
| `recommend_dresscode` | 구조화 상황 → 권장수준 + 추천/금기 + 면책 (결정적 규칙) | 동일 |
| `create_tpo_guide` | 권장수준 → 가이드 카드(자기완결 `check_criteria` 포함) | 동일 |
| `check_outfit` | 카드 기준 + 착장 텍스트 → 부합/조정/부적합 + 이유 | 동일 |

**공유 타입(`models.py`)**: `occasion`, `dresscode_result`, `guide_card`, `check_criteria` — FastMCP가 타입힌트로 스키마 자동 생성. stateless라 도구 간 값 전달은 클라이언트가 담당하므로 모양만 맞춰두면 됨.

## 3. 역할 (3인 중 1인 연수)

### 🅰️ Dev A — 계약/코어 (잔류)
- `models.py` 공유 타입 **1인 소유**
- `parse_occasion`, `recommend_dresscode`, `rules.py`(규칙 테이블)
- `privacy.py`(MVP 최소) · `tests/conftest.py` + e2e 체인 테스트 (연수자 부재로 흡수)

### 🅱️ Dev B — 파이프라인/플랫폼/배포 (잔류, **임계경로**)
- `create_tpo_guide`, `check_outfit`(연수자 부재로 흡수)
- F3 시나리오 데이터셋(6종) + 면책/계절 카피
- **`server.py` + 카카오 클라우드 배포 + PlayMCP 등록 + Inspector 검수** ← 데드라인 임계경로
- `register_tools`, 버전 3중 동기화, CI

### 🅲️ 연수자 — v1/매칭 (6/28~7/7 부재, 7/8 복귀)
- **복귀 후**: `extract_color`, `recommend_bottoms`, `get_dresscode_rules` + 장례·종교 시나리오(3인 크로스리뷰)
- 컨테스트 제출물 아님 → 복귀 후 첫 임무는 **빠꾸 대응 백업 + 라이브 서버 신선한 눈 QA**

> **충돌 방지**: `register_tools`·`conftest.py`·`server.py`·`models.py`는 각각 단독 소유자 지정. 나머지는 자기 도구 모듈 + 자기 테스트만 추가.
>
> **부하 경고**: B가 임계경로(배포·등록) + 도구 2개를 짊어짐. 빡세면 **A가 F3 카피·테스트를 흡수**해 B를 배포·검수에 집중시킬 것.

## 4. 핵심 전략: 배포·등록 "조기 검증"

가장 큰 리스크 = 첫 카카오 클라우드 배포 + PlayMCP 등록에서 터지는 문제. 7/9에 처음 하면 빠꾸 버퍼를 인프라 디버깅에 소진.

→ **7/1~2에 현재 스켈레톤(example 도구)으로 배포+등록 파이프라인을 한 번 끝까지 통과**시킨다. 그러면 7/9은 "첫 배포"가 아니라 **"내용 업데이트 재제출"** 이 되어 리스크 급감.

> 같은 7/1 검증에 **이미지(사진) 지원 여부**도 함께 확인한다(`echo_probe` 프로브 도구). 사진 입력의 MCP 구조적 제약·옵션·검증 절차는 [image-input.md](image-input.md) 참고 — `check_outfit` 입력 스키마가 여기서 갈린다.
